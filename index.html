核心定义

我们卖的不是 AI，
而是职业能力的结构化交付

用户真正买的，从来不是模型、不是 prompt、不是聊天机器人。

01

更快完成工作

把原本需要大量认知负担的职业任务，转化为可即时调用的结构化流程，让每一次执行都有迹可循。

最小单元

平台真正的最小单位，
不是职业，而是 Skill Unit

不先定义职业目录，先定义什么任务适合被 skill 化。每个 Skill Unit 是一套完整的可执行规格。

SkillUnit Schema · example: legal_contract_review ● ACTIVE
task_name 合同风险点审查

法务专员 合规顾问 律师助理

判断标准

不是所有工作
都值得做成 Skill

在动手之前，先回答这五个问题。每个指标都是过滤器，不是加分项。

01

是否高频

每周至少发生一次的任务才有 skill 化价值。低频任务的边际成本过高。

02

痛点是否明显

用户是否真的为这个任务感到费力、焦虑、耗时？轻微不便不足以成为产品动机。

03

输入输出是否标准

任务的输入材料和预期输出是否有规律可循？完全开放式的任务难以 skill 化。

04

用户愿意直接采用结果

如果用户每次都需要大幅修改输出，说明 skill 的封装质量或定义方向存在问题。

05

风险是否可控

对于高风险输出，是否有明确的人工介入节点？不可控风险的任务需要特殊架构。

做的是职业工作流平台，不是职业聊天机器人。聊天是入口，skill 执行才是核心价值交付。

结构化能力是核心资产

核心资产是 skill 结构化能力，不是职业标签或 prompt 集合。能力在于如何定义和编排，而非内容本身。

03

共享骨架上的专属适配

个性化方式是共享骨架上的专属参数，不是全定制开发。规模化与专属化不矛盾，关键在于架构设计。
